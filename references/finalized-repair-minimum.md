# Finalized Repair Minimum

Use this file only when a controller or user explicitly starts a finalized repair checkpoint from a copied bounded draft in a `finalized/` directory. It is the generator-facing minimum path. Do not load `validation-protocol.md`, `development-log.md`, full style-profile reports, checker source, tests, or hidden thresholds while repairing.

## Contract

- Work only on the copied `finalized/draft.md`.
- Freeze the selected genre before editing. The controller passes that genre to gates; the repair agent only uses the genre named in `repair-brief.txt` when present.
- A repaired article exists only after the complete article is written back to `draft.md`.
- A nonzero `--strict --repair-brief` exit usually means not passed, not a broken tool.
- Do not print a proposed final article to the terminal and keep thinking.
- Do not tune ratios one by one. Use the brief to choose one source-level rewrite.
- Do not rediscover this skill by globbing, `Test-Path`, listing parent skill directories, searching sibling skills, or reasoning aloud about how to find the skill directory. The repair attempt is artifact-only: read `draft.md`, read `repair-brief.txt` when it exists, use this minimum path, write one complete revised artifact, and let the controller validate. An unverified artifact is better evidence than no artifact or a path-search trace.
- Do not use TODO tools, checklist panels, plans, or long diagnostic narration as the repair artifact. The next useful action after a not-pass brief is reading the current `draft.md`, rewriting the whole article from one source action, and writing it back to `draft.md`.
- In formal finalized checkpoints, this file is not an open-ended ordinary-user repair loop. The repair agent gets one complete source rewrite after the not-pass brief, writes the artifact, and stops. Post-write validation belongs to the controller, not to the repair agent.
- A second `Write draft.md` or `Edit draft.md` in the same finalized attempt is invalid controller evidence. After writing `draft.md`, stop on artifact persisted. If a chat response is required, output only `artifact_written`. Do not run any checker or controller-helper command, do not calculate per-thousand thresholds, do not argue that checker metrics conflict, and do not write another version.
- Do not run `check_anlin_violations.py`, `check_style_profile.py`, `clean_run_checker.py`, `prepare_finalized_repair_brief.py`, `python -c`, `Measure-Object`, `wc`, `Get-Content` length probes, regex counters, line-length distributions, connector counters, path probes, source/test reads, threshold searches, or old-log searches during the repair attempt. A visible body-shape check happens before the write; exact validation is controller work and invalidates the repair attempt as metric-chasing evidence.
- The single write is atomic. If you notice a typo, a compressed prose block, a missed hard-gate issue, or a line-shape problem after writing, do not patch it with `Edit draft.md`; stop and let the controller rerun gates and record unresolved repair evidence.
- Because the single write is atomic, do a visible body-shape check before saving. Do not write a standard diary that is mostly one short sentence per row, has no moving long action/speech/thought rows, has 45-70 short caption rows with zero true long rows, has 75-80+ body lines, has almost every line trailing with `，` and almost no landed `。`, or has only 8-25 dense prose rows. A true long row is not a 10-18-character caption; it should visibly carry a hand, reply, payment, body, speech, or thought movement beyond about 24 Chinese characters. Merge and rebuild the action clusters before the write; after the write it is too late to patch without invalidating the finalized attempt.

## Repair Brief Input

The controller may create `repair-brief.txt` before the repair attempt with `scripts/prepare_finalized_repair_brief.py`. That file is the public repair interface. It can include hard-gate blockers and a compact `check_style_profile.py --repair-brief` block, but it should not include the full metric table. The repair agent consumes the file; it does not generate the file.

From inside the `finalized/` directory, the repair-agent file set is deliberately small:

- `draft.md`
- `repair-brief.txt` when present
- this `finalized-repair-minimum.md` when already available through the triggered skill

If `repair-brief.txt` is absent, do not search for checker scripts or ask how to run gates. Read `draft.md`, use this minimum path, write one complete revised `draft.md`, and stop. The controller records local validation as unresolved and reruns formal gates after the artifact is frozen.

After the artifact is frozen, the controller runs the strict hard gate, full style-profile report, overlap checks, trace audit, and controller summary. The repair agent should not use those full reports as its writing interface.

If the repair brief says `formal_gate: not_pass`, shows `checkpoint_pass: false`, or lists hard-gate blockers, treat that as normal repair input. Do not open the full profile report, do not calculate thresholds, and do not continue discussing findings. Read `draft.md`, apply one primary source rewrite, write the complete article back to `draft.md`, then stop. The controller reruns gates after the artifact is frozen; the repair agent must not run any gate loop or make a second metric-shaped edit.

## Repair Choice

If the hard gate has blocking errors or warnings, fix that source function first and write a complete `draft.md`. The profile brief is the rhythm/texture interface for the same rewrite; it must not make the repair agent forget a hard-gate finding such as unsupported facts, rough self-damage shortage, period grid, prompt leakage, or formulaic message glue. Do not defend the draft or argue with the checker.

If a standard-diary hard gate reports `粗粝自毁信号不足`, do not add a curse, shame label, or isolated dirty object. Replace one private room/screen/ledger packet with a low consequence that changes the next action while pressure is still active: a payment or route action stalls, somebody waits or asks a plain question, a wet/dirty hand changes the reply, a doorway/body problem exposes the narrator, or a bad reply makes the next move smaller. Room-only water, wet sleeves, wet pants, wet slippers, basins, hot-water noise, opening the door, seeing hallway light, or seeing a neighbor light under the door are still private/threshold texture unless they force another person, payment, route, reply, or body action to change before the scene retreats. This is also the safest way to repair period grids: the ugly consequence creates unfinished action lines, rough longer rows, and short failure drops without tuning punctuation.

A public hinge is not the same as a cameo. A neighbor, cashier, rider, or other person must touch the low fact in the same action chain: they wait while the narrator cannot pay, ask about the wet/dirty hand, step around the stain, see the failed reply, block the door, or force the object/body/payment problem to change the next move. If the other person only asks an informational question while the wet sleeve, wet pants, balance screen, or room noise stays private, the repair has not left the room yet. Door threshold imagery alone does not count: opening the door, seeing corridor light, noticing a neighbor light, and closing the door fast are still private texture unless the contact or practical handoff changes the next action.

If the hard gate is clean but the repair brief says `formal_gate: not_pass`, choose exactly one source rewrite before controller validation:

- `punctuation` or `line_rhythm`: rebuild page shape first, not metric surface. A standard repair should usually become 6-8 visible breathing clusters and about 45-70 actual body lines, with rough long action/speech/thought rows, earned short failure drops, and mixed comma/hard-stop landings. Add short drops only when an action lands badly. Do not globally merge rows into comma chains and do not split the draft into tiny rows. For standard diary, do not shrink the repair below about 900 body Chinese characters or turn it into 70+ similar short rows; keep several rough long rows inside the clusters. A 45-70-line corridor is not a target count: if most rows are equal 10-20 character sentence units, merge and rebuild the action instead of calling it line-broken. If the planned rewrite has zero obvious long rows, if only one or two rows run long enough to carry speech/action/thought movement, if no row visibly exceeds about 24 Chinese characters, or if the body still has only 8-25 dense prose rows, it is not ready to write. A repair like 53 short rows and 580 body characters is still a caption grid, not a standard diary. Line-final comma is not the goal either: if nearly every line trails with `，` and the draft has almost no landed `。`, restore hard stops where actions actually land before saving. If the rewrite has become 650-899 body Chinese characters after starting from a full standard draft, it has lost source mass; add one active consequence cluster rather than saving a shorter clean-looking article.
- `length` or prose compression: keep the article visibly line-broken while rewriting. A standard repair should not become 8-15 long paragraphs or a few very long rows. If the source move needs more mass, add one active consequence cluster and preserve broken-line breathing; do not compress the whole draft into paragraph prose to avoid a period grid.
- `connectors`: change what happens next so a turn is needed: failed reply, payment handoff, body interruption, route/object change, or another person's plain line. Do not paste connector words.
- `texture` or `ngram_texture`: delete one repeated body, screen, room, money, route, food, or object packet. Replace its function with a social or practical consequence that changes reply, payment, route, room position, body state, or next action.
- `structure` or `cognitive_mechanism`: cut prompt echoes, decorative proof details, and explicit cognition. Let a concrete object lead to a crooked read, a reality puncture, a defensive recovery, and a practical exit through action rather than explanation.

When multiple families appear, do not make one patch per family. Use the brief's primary source rewrite as the top action. In social-decline drafts, prefer rebuilding the refusal aftermath and rhythm together: remove one room/object proof packet, make the post-refusal response or payment/route/body problem change a later action, and rewrite the visible body as breathing clusters. Do not repair social-decline message surfaces by enumerating message order: avoid `X发了第二条`, `第二条只...`, `又发来一条`, `下面还有一条`, `下面还有一个...`, or `后面跟着...` as plot glue. Use one cropped screen surface, address preview, voice bar, unread mark, hand/body interruption, or reply consequence instead. If the draft can still be summarized as `message -> calculation -> refusal -> he said OK -> room object`, the refusal has not become a source engine; rebuild from the reply aftermath before writing. A plain OK reply plus a private screen mark is still a screen loop unless it changes a visible next action. That source move can address punctuation, line rhythm, connectors, texture, and structure at once.

Do not repair that screen loop by making the social consequence larger than the scene. `第二天群里有人问`, `有人@我`, `你怎么说项目忙`, `项目不是结了吗`, `正在输入`, and similar group-chat public exposure are usually another screen loop, only with a crowd. Cut them before the atomic write. Use one local consequence instead: a smaller reply, a payment/route action that will not finish, one person waiting or asking while the phone/payment/reply is still in hand, a dirty/wet hand changing how the answer is sent, or a door/body/object problem that lowers the narrator in the same action chain. The consequence must depend on the refusal. An unrelated delivery, food burn, room chore, or outside errand after the `OK` reply is not enough; if deleting the wedding invitation leaves the scene intact, it is off-axis residue, not social-decline repair.

Do not repair by moral etiquette. A narrator red packet with `抱歉人不到钱到`, followed by `人不到没事`, `下次一起吃饭`, `心意到了`, or a neat `好` reply, closes the refusal too cleanly and often causes a literary tail. If money remains, keep it as a stalled or awkward action; if the other person answers, let it shrink the next move rather than forgive the story. The tail should leave a practical residue, not a completed social settlement.

If a social-decline repair has already gained a valid low-status/public hinge, do not add more roughness just because an older run reported weak engine. Recheck the current hard gate first, then repair the remaining source failure. If the current failure is comma-drag or period-grid, rebuild only the clusters where the refusal aftermath is still active: edited reply, unfinished transfer, route hesitation, door/body interruption, ordinary response, or a dirty/wet hand changing the phone action. Do not turn this into all-comma breath or all-period captions.

If the copied draft or the planned repair is a dense 8-25-row prose article, treat that as the first source failure even when some social-decline content has improved. Do not save a 15-line "better story" and hope the controller accepts it. First convert the refusal aftermath and side action into actual broken body rows: one line continues because a hand/reply/payment/body action has not finished, one longer row carries the awkward movement, and one short row lands a bad answer or low-status fact. Repeat this across several clusters before the atomic write.

## Stop Rule

After writing the revised `draft.md`, stop. If controller validation then finds the same or opposite failures, such as period grid -> comma drag, comma drag -> period grid, too few short drops -> tiny-row grid, texture thinning -> underbuilt source, or refusal consequence -> room texture, record unresolved finalized repair. The next fix belongs in the skill architecture or first-draft source guidance, not in another metric-shaped rewrite by the same repair agent.

If the revised article is only printed in a log/chat but `draft.md` is unchanged, mark artifact failure. If `draft.md` changes but the style gate is still `review` or `revise`, mark finalized unresolved or failed according to the controller summary; do not call it ready for blind rounds.
