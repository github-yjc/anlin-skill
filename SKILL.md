# anlin-writing

This skill generates, reviews, and evaluates Anlin-style Chinese prose for anonymous blind evaluation.

The goal is narrow:

> Make generated text less stably identifiable as generated text in anonymous blind review. Report only test conditions, sample size, and recognition/pass rates. Do not claim real authorship, provenance, or "indistinguishable from the original author."

Generated prose should not contain process labels such as "仿写", "AI", "生成", "验证通过", or "38篇语料". Controller notes and validation reports may mention methodology; the prose body must not.

This skill is self-contained at runtime. Do not require or call any external anti-slop, author-style, or personal writing skill while generating, repairing, or judging. The anti-AI-writing guidance needed for this task lives in `references/anti-ai-slop.md`, `references/feature-budget.md`, and the checker bundled in this skill. External writing skills may be used only by the developer while improving this skill, not by a generation agent using it.

## Modes And Artifact Locations

Use the smallest mode that matches the user's request:

- **Ordinary user mode**: the user wants an article, revision, or saved prose. Ask only for missing information that materially changes the result. Use the checker and repair loops as needed to clear hard errors and satisfy the user, but do not mechanically chase every ratio warning. If the same issue repeats, change the scene source, title choice, background specificity, or rhythm model instead of piling on local patches.
- **Clean-eval mode**: the user or controller is testing whether the skill naturally guides a fresh agent. The generator receives only a realistic user prompt plus this skill; the controller may also place an empty `.anlin-clean-eval-mode` marker in the task workspace to select this mode without adding style hints. It may call `clean_run_checker.py` at most twice and must output the current `draft.md` after the second call, even if problems remain. This mode measures the source guidance; it is intentionally stricter than ordinary use.
- **Finalized repair mode**: the controller is testing whether the same draft can converge after ordinary multi-round repair. Start from a copy of the bounded draft in a separate `finalized/` directory. Freeze the selected genre from the prompt, case name, or bounded draft before editing; a sincere/micro-hope repair must not drift into standard diary just because global profile ratios look safer. The repaired article must be written back to that directory's `draft.md`; terminal-only "final version" prose does not count. Normal checker success is not enough; the finalized article must be validated by strict hard gate and style-profile audit before it can count as passed. `review` is still unresolved; only `pass` means the final article is clean enough to narrow the next change mainly to source guidance.
- **Controller validation mode**: after an article exists, run corpus comparison, style-profile audit, blind rounds, placebo rounds, and reports. Do not feed controller findings back to a fresh clean-eval generator as hidden writing advice.

For development testing, treat clean-eval and finalized repair as two separate freeze points. Clean-eval freezes the fresh-agent result after bounded preflight and limited checker-driven repair; the wrapper also records hidden stage snapshots for the controller: first submission, checker-call submissions, and bounded final. These snapshots are evidence only, not writing advice for the generator. Finalized repair freezes a copied draft after ordinary multi-round repair. If finalized passes but clean-eval fails, improve the earliest generation guidance and clean-eval repair instructions. If finalized also fails or remains `review`, inspect the broader skill architecture, fact gates, repair path, profile assumptions, and checker behavior before adding more local detector rules.

Do not write generated articles into the skill directory. Use these locations:

- If the user only wants prose in chat, write temporary `draft.md` in the current task working directory, run checks there, then output only the article.
- If the user asks to save a file and no output directory is given, ask for an output directory in the intake turn. If the user does not answer, save in the current task working directory, not inside `<skill-dir>`.
- If the current working directory is the skill directory, create or use a task/eval workspace outside the distributable skill before writing `draft.md`.
- For tests and blind review, use an external evaluation workspace such as `<eval-workspace>/iteration-<n>/eval-<id>/draft.md` and `<eval-workspace>/iteration-<n>/benchmark.json`. The `evals/` folder defines prompts and protocols; it is not the default output sink.

When the task workspace is already set by the controller or current shell, write and read the article using the relative path `draft.md` or `.\draft.md`. When using a write/file tool, the file/path argument must be exactly `draft.md` or `.\draft.md`; never pass an absolute path, never pass a date-stamped path, and never pass `<skill-dir>/iteration-.../draft.md`. In clean-eval mode, confirm the current directory with `Get-Location` / `pwd` before the first write even if the marker check succeeded. Never substitute a different date-stamped directory or a path from another run.

`<skill-dir>` is only for resolving bundled references and scripts. It is not an output root. Never append the evaluation workspace name, iteration directory, case id, or `draft.md` under `<skill-dir>`. If `Get-Location` shows `<skill-dir>` or a path ending in `anlin-writing`, do not write; switch to an external task/eval workspace or stop the clean-eval run as invalid. If `Get-Location` shows the case workspace, write exactly `draft.md` there.

If the current task workspace contains `.anlin-clean-eval-mode`, treat the request as clean-eval mode even if the user prompt itself reads like an ordinary article request. The marker is a mode selector, not writing advice; do not mention it in the article.

At task start, check whether the current task workspace contains `.anlin-clean-eval-mode` before writing any draft or running any checker. For any request that will write `draft.md`, the first tool action should be a lightweight current-directory marker check such as `Test-Path .anlin-clean-eval-mode`, `Get-ChildItem -Force .anlin-clean-eval-mode`, or equivalent. In clean-eval mode, run `Get-Location` / `pwd` before the first draft write and verify it is the external case workspace, not `<skill-dir>`. A first `Write draft.md` before this marker check is a contaminated clean-eval run when the marker exists, and a first write without current-directory confirmation is not a valid bounded run. In clean-eval mode, never call `check_anlin_violations.py` directly in the bounded case directory; the only checker entrypoint is `clean_run_checker.py`.

## Clean-Eval Minimum

When the user asks for a complete Anlin/日寄 article, keep this minimum protocol in working memory even if no extra references are opened:

1. First `draft.md` must already be a complete article with title, about 950-1150 body Chinese characters, roughly 45-70 body lines, mixed short and long lines, and several early comma-ended continuation lines. Do not write 8-15 giant prose lines, 100 tiny lines, or an overfilled 1350+ character article that stacks body/screen/object details just to pass checks. A draft under 900 body Chinese characters is a strong incomplete-standard-diary risk. A 900-949 character draft is a boundary case, not automatically wrong; it becomes an underbuilt source shape when paired with 45-75 similar short rows, almost no long action/speech/thought lines, weak connectors, no rough middle engine, or no ugly body/social turn. Rebuild that from the source loop before saving; do not pad it with a few extra symptoms or captions.
2. Before the first checker, the body must already have a real engine and a middle rhythm corridor: an ugly self-own/body/social cut, a crooked misread or system joke, an off-axis daily consequence, several different natural small connectors from `其实/觉得/发现/好像/不过/突然/于是/因为/所以`, a few true short breath drops around 8 Chinese characters or fewer, and several rough longer action/speech/thought lines. In formal clean-eval, three connector families is a thin boundary; aim for four or more natural turn traces when the article has enough movement. These are movement signals, not content quotas. A connector should appear because the narrator changes action, corrects himself, finds a wrong thing, delays a reply, gets interrupted, or moves to another consequence; do not solve it by repeating `其实/已经/当时` as glue or by adding every possible texture family. Each concrete noun must change action, social position, body consequence, or the next scene. Do not write a prose-paragraph article first and then rely on scripts to cut it into shape; the first persisted clean-eval draft is already part of the measurement.
   Treat a 10-15 paragraph article that is later split by scripts as a failed source start even if the final line count becomes acceptable. The first saved article should already have breathing clusters: some lines run through `，`, some land with `。`, a few short drops carry a bad reply/body lowering/failed decision, and several rough longer lines carry action or speech. Count actual visible body rows before saving; do not trust a mental claim that the draft is "about 60 lines." Do not create short drops by chopping every sentence into captions.
   Short sincere/micro-hope has the same source risk in smaller form. Do not write 35-55 closed sentence rows where nearly every line is a complete `。` sentence, then patch a few commas at the beginning. Build 4-7 uneven clusters before saving: one action or reply can trail into the next line with `，`, one longer clumsy line carries memory or logistics, and one short line lands a fact-retreat. Avoid using `后来/已经/当时` as the main way to stitch memory order; let a dragged slipper, a wet sleeve, a wrong reply, a bowl, a door, or another low object move the piece.
   A near-miss first draft is still not ready. If it has only 35-44 actual body lines, or roughly 850-949 Chinese characters with only two connector families, add one full off-axis life cluster before writing `draft.md`: 6-10 visible lines where a body interruption, neighbor/cashier/roommate line, small errand, dirty object, payment action, or screen residue changes the next action and naturally creates new turn traces such as `发现/不过/突然/于是/因为/所以`. Do not patch this with one explanatory paragraph, one extra symptom, or a decorative short drop.
   A rough self-own is not a polite line saying the face probably looks bad or the narrator seems stupid. It must become an outside consequence: dirty clothing noticed, body noise/cost, bad reply, mistaken identity, low-status practical maneuver, or a body failure that changes the next action. Ordinary unclean consequences such as wrong slippers, snot, dirty pants seen by someone, sticky drain hair, worn-through socks, a public stomach noise, or someone reacting to the narrator's dirty hand can work when they change action or social position; they are not required ingredients.
   For quiet interior prompts built from phone/feed + room + food/order + bed, do not let the first draft stay inside private observation. Before writing `draft.md`, choose one outside contact or practical handoff that leaves residue in the room/body/action: a delivery handoff, neighbor/cashier line, door/stairwell interruption, leaking package, wrong reply, or visible stain. A stranger looking once, a mirror face, or a line saying "I looked bad" is still too soft unless it changes the next action.
   For illness, pain, gout, stomach, urine, tooth, sweat, or insomnia prompts, choose the first outside consequence before writing the first body line. Do not build a private case report: symptom list -> search result -> food taboo -> refrigerator inventory -> room smell -> ambient sound. Let one body symptom interrupt action, then leave through a practical or social consequence such as a failed walk that changes a route, spill, bad reply, doorway/neighbor/cashier exposure, shopkeeper seeing the limp, bathroom/slipper problem, payment/key trouble, or small errand. A phone message is still too private unless it changes a visible reply, route, payment, doorway, or next-day problem. Do not make that consequence an unsupported current-work chain: no invented leader, office file, Monday deadline, leave penalty, attendance, salary deduction, work group, or current shift unless the user supplied it. Convert searched claims like `有人说...` into one cropped screen phrase or article title; do not keep `有人说/另一条说` chains. If the same symptom, body part, food object, room smell, ambient sound, and diagnosis label keep returning, cut one whole packet before adding any new material.
   Do not answer weak-engine or roughness warnings by stacking the same body/object packet. If hands, dirt, fruit/food color, cutting board, garbage, heat, payment, screen, room darkness, fan noise, or air-conditioner noise keep returning, cut one whole packet and replace its function with a social or practical consequence that changes a reply, payment, route, room position, or next action. Private grime and ambient sound are texture; exposed grime or sound-driven interruption becomes movement only when another person, payment action, failed reply, body demand, or practical maneuver changes the scene.
   A stranger/vendor/social-comparison scene should not become a neat transcript where the other person recites the prompt. Keep speech embedded without theatrical `“”` / `""` when possible, then exit through payment, object handling, dirty hands, body noise, route, or silence. Also avoid identity-probe dialogue such as `你是不是以前那个大学的，就是...` when it mainly delivers school/place/career/salary facts; lower it to a rough side remark and let the practical residue carry the damage. Remove polished caption metaphors such as `那句话像一颗钉子` or `消息像扔进井里`; let the object failure, body reaction, or next action carry that pressure.
   Avoid ordinary-looking negation repairs that still form an AI binary reframe. `也不是疼，就是...`, `其实不是想X，就是...`, `不是认识，就是...`, `不是因为X，而是因为Y`, `不是X。是Y`, `我不是叔叔，我只是...`, and `当时最疼的不是X，是Y` should be rewritten as positive physical facts, dialogue, or action consequence before the first checker sees them. Do not confuse this with ordinary `是不是` questions; those are allowed when they are real social movement, but become unsafe when they make a stranger deliver the assignment's identity facts. Scan the whole article for all occurrences; generated drafts often fix the first obvious one and leave a softer second one. If only this surface remains, replace locally and do not add scenes, short drops, money lines, body symptoms, app details, or other texture.
3. Treat `references/anti-ai-slop.md` as negative constraint only, never as subject material. If the user did not explicitly ask for AI/GPT/model/text-detection content, the article body should contain zero AI/GPT/生成文本/识别AI scenes. Do not write about AI writing, AI detection, AI chat windows, model output, or article-detection advice merely because this skill discusses those risks.
4. Before writing `draft.md`, remove binary reframe sentences and group-chat/comment-chain surfaces such as `有人发/有人说/又有人/底下有人/另一个说`; convert chat pressure into one screen/action/body consequence instead. Treat social-feed prompts the same way: 朋友圈、短视频、评论区、点赞、礼物照, 年度总结, 新年 flag, and转场视频 are pressure surfaces, not a scene list. Keep at most one cropped visible surface before the first checker, then make it change the narrator's body, reply, payment, room, route, or next action. Do not stack three posts, cancel-like actions, captions, and a final latest video just to prove the prompt. For cross-year, old-chat, memory, or annual-summary prompts, do not write a screen-archaeology chain such as feed -> old chat -> group chat -> contacts -> app push. Let one old record trigger one physical/practical consequence, then leave the phone through a body, room, route, food, payment, or outside interruption. Do not print the article to chat before validation. First write the complete candidate to `draft.md` in the task workspace, run the checker flow for the selected mode, then read `draft.md` once and output that exact content. A visible pre-check article followed by tools contaminates a clean-eval run. If `.anlin-clean-eval-mode` exists, the intended tool order is marker check -> load `references/clean-generation-brief.md` -> write one complete `draft.md` -> run `clean_run_checker.py`; do not compose trial paragraphs, repeated scratch drafts, or scene plans in the terminal while looking for the right article.
   For old-chat, annual-summary, archive, graduation-memory, or cross-year prompts, decide the present consequence before writing: a wrong reply, delivery/cashier/neighbor interruption, body noise, leaking package, wrong slipper, route/payment problem, or unfinished chore. The old record is a trigger, not the engine. If it only leads to quiet comparison plus room objects, replace one private object packet with a present action that lowers the narrator or changes the next scene.
   A complete article is not a complete inventory of prompt nouns. For multi-item prompts such as discount shopping, social feed, parcel pickup, wrong-size goods, family pressure, route, and screen notifications, pick one lived failure as the driver and let only one or two prompt surfaces interrupt it. Bury or discard the rest unless it changes body, money, route, reply, room, or social position. Do not write one neat scene for every supplied noun in the user's order.
   Explicit user prohibitions are hard source constraints. If the prompt says not to write money, consumption, prices, games, romance, family, or any other domain, remove that whole domain before drafting instead of keeping a softer neighboring version. `不要写金钱、消费或价格` means no store checkout, buying water, receipt, payment gesture, balance, discount, amount, delivery order, or price arithmetic; a prompt-supplied red packet can remain only as a cropped screen surface if needed, not as spending or calculation.
   Do not print an English or Chinese scene plan or metric TODO list between tool calls. Text such as `Let me plan`, `Scene 1`, `The preflight says`, `I need to`, or visible checker-metric restatement is process chatter. Keep it private; after the checker flow, the only visible answer should be the exact `draft.md` article.
5. In clean-eval mode, run `scripts/clean_run_checker.py draft.md --strict --draft-gate`. It delegates to the normal checker and enforces the two-call limit. Calling this wrapper commits the current `draft.md` to clean-eval protocol; do not later reinterpret the same run as ordinary user mode. If it prints `CLEAN_RUN_PREFLIGHT`, the draft was not ready and no checker call was consumed; revise from the preflight message and the already-loaded references, then run the wrapper again. If it says `body_lines < 45` or `prose_block_shape=compressed`, first run `scripts/split_long_lines.py draft.md --in-place --target-lines 58` or `scripts/rebalance_line_rhythm.py draft.md --in-place`, inspect once, then repair content; if it says `body_lines > 90`, `short_line_grid`, `medium_short_line_grid`, or `long_lines < 4`, run `scripts/merge_short_lines.py draft.md --in-place --target-lines 68` or `scripts/rebalance_line_rhythm.py draft.md --in-place`, then manually preserve several rough longer action/speech/thought rows. If it says the source shape is underbuilt, rewrite from the source loop before overwriting `draft.md`: keep only useful facts, add one off-axis consequence and one rough body/social turn, then rebuild the middle corridor. For short sincere repair, do not extend by repeating the same phone/message/bowl/water/oil packet; delete one repeated packet or restart from a different present action. Do not rewrite the same prose as 100+ tiny rows or the opposite 30-40-line prose block. Do not create line breaks by deleting punctuation. Do not call `check_anlin_violations.py` directly, grep/read checker source, or browse tests to discover hidden tokens after a preflight. If it prints `CLEAN_RUN_PREFLIGHT_STOP`, the draft stayed unready after the bounded preflight attempts; stop repair work, read `draft.md` once, and output it unchanged so the controller can mark the clean-eval run invalid or failed. Do not switch to the normal checker after this. If the first actual call reports prose compression, short-line grid, uniform rhythm, too few long lines, or opposite rhythm drift after a script, run `scripts/rebalance_line_rhythm.py draft.md --in-place` and inspect once. If it reports `行末逗号比例`, run `scripts/soften_line_endings.py draft.md --in-place`. Then repair content once.
   `CLEAN_RUN_PREFLIGHT_STOP` and `CLEAN_RUN_STOP` are protocol stop signals, not command failures. They are final boundaries for the bounded case: do not write `draft.md`, do not repair, do not run the normal checker, and do not run a third command sequence. The next tool action must be reading `draft.md` once and outputting it unchanged. The wrapper may exit successfully at those boundaries so the agent does not treat the stop as a shell failure. Success exit code does not mean the article passed; the controller reads the state file and summary report.
6. If you write `draft.md` again after any rhythm script, the previous rhythm repair no longer counts. Do not paste a new 8-15 paragraph version over the repaired file and then call the checker as if the old split/merge still applies. Either rewrite inside the current line-broken corridor, or rerun `rebalance_line_rhythm.py` / `soften_line_endings.py` before the next wrapper call. Before the second checker, run `rebalance_line_rhythm.py` when the current file has fewer than 45 body lines, more than 75 body lines, fewer than ten natural long lines, visible long prose blocks, a short-line grid, or 45-70 neat medium rows with no true short breath drops. Run `soften_line_endings.py` again when the first 20 body lines mostly end in `。`. The target is a middle rhythm corridor, not perfect ratios.
7. Do not run the second checker until the current `draft.md` is the version you are ready to output unchanged. After the second checker, the next and only tool action may be reading `draft.md`; no third write, no third checker, no manual final repair, even if errors remain.
8. If the first actual checker reports `背景展示堆砌`, delete at least one whole background family before the second checker. Usually cut game or AI/platform first unless that family is the user's actual prompt. Do not keep 王者/AI/知乎/外卖/我妈/脚踝 together merely because each fact is individually supported.

| Post-draft critique | `references/review-rubric.md`, `references/writing-checklist.md`, `references/self-check.md` |
| Anti-copying and overlap checks | `references/anti-pastiche.md` |
| Corpus-prior ratio audit | `references/stylometric-ratio-protocol.md`, then `scripts/build_style_profile.py` / `scripts/check_style_profile.py` |
| Full validation and blind review | `references/validation-protocol.md`, `references/blind-judge-angles.md` |

Do not read every reference before drafting. The generation pass uses a small state model; the critique pass uses the rule library.

Do not compensate for weak prose by loading another style or anti-AI skill. If the draft smells synthetic, revise through this skill's own lenses: scene-first causality, prompt displacement, background restraint, rhythm variance from action, and concrete body/social/material consequence.

Do not use corpus ratio targets as a pre-draft recipe. `references/stylometric-ratio-protocol.md` and the style-profile scripts are for post-draft audit, controller validation, and targeted repair. If a ratio drifts, repair the underlying function: length through lived action, line rhythm through interruption/speech/body, connector drift through less explanatory glue, and title drift through `references/title-model.md` rather than a universal default.

For clean-eval generation, checker call count is a hard protocol boundary. Checker findings are not tool failures, even when the checker exits nonzero or reports many `error` findings. The wrapper may create `.anlin-clean-run-snapshots/` beside `draft.md`; do not read or use those snapshots while generating. They are controller evidence for separating first-draft guidance from checker-driven repair. After the second checker call, the only allowed tool action is reading `draft.md` once to output it exactly. Do not edit, write, compare, manually repair, explain the checker result, invoke fallback rules, or run another checker after it. A third checker call, post-second write, or unpersisted final repair contaminates the test more severely than leaving draft errors unresolved. The validation controller will decide pass/fail.

For ordinary user mode, checker loops are not limited to two calls. Continue repairing hard errors, process leakage, unsupported facts, missing title, copied phrasing, and obvious AI-slop until the output is usable or the user stops. Still avoid infinite metric chasing: when warnings repeat, repair the underlying scene function rather than adding labels, background facts, punctuation, or checklist features.

For clean-eval generation, the visible final article must be exactly the current `draft.md` content after the last write. Do not create an unpersisted "manual repair" in the final response after the second checker call. If the second checker still reports errors, output the current `draft.md` anyway and let the controller fail it. The fallback for file/checker execution failure applies only when the command cannot run or the file cannot be written; it never applies to ordinary checker findings.

## Workflow

### 1. Confirm Inputs

If the user has not already supplied enough context, run one intake turn. One intake turn means one grouped message containing only the missing questions that materially affect the result; it is not one single question, and it is not permission to interrogate the user through multiple back-and-forth rounds.

First extract what is already known from the user's request. Do not ask for information that is already clear. If the request is only "use anlinwriting/anlin-writing to write like Anlin", ask only the missing items from this list:

- target date/time or `inferred`
- genre: standard diary, sincere, micro-hope, surreal, mixed, or unspecified
- concrete material: one recent small event, body state, social pressure, money/food/route/screen/family detail, or `none`
- output mode: chat only, or save to a user-provided output directory
- whether low-confidence fictional/projection output is allowed when background is missing
- whether web/background lookup is allowed

If date, background, genre, and output mode are already clear, do not ask. If the user refuses, leaves inputs open, or does not answer in a non-interactive run, proceed with `inferred` or `fictional` status, use lower specificity, and avoid specific real-world events, named places, current company facts, current game-match details, or other unsupported claims.

### 2. Build The State Card

Route genre before applying article length, rhythm, and repair targets. `日寄` / broad daily montage / several daily pressure surfaces usually means standard diary. Mother's Day, direct thanks/apology, companionship, or a family memory that would be cheapened by forced jokes can route to sincere mode. It can also be sincere when the mother figure is carried indirectly through `她` plus holiday/no-message pressure and concrete care logistics such as food sent home, `吃了吗`, `还有没有菜`, `天冷了多穿点`, or a bag tied by hand; the article does not need to say `妈妈` to earn the route. Small acceptance can route to micro-hope; loosened causality can route to surreal only when the scene earns it. Ordinary family pressure, family dinner, or one `我妈` scene can still be standard diary.

For clean-eval generation, build the source state from `references/clean-generation-brief.md` only: friction -> pressure displacement -> off-axis consequence -> fact gate -> complete titled draft. Do not open `runtime-brief.md` before the first complete `draft.md`.

For ordinary non-formal drafting, or for repair after the first checker pass, form an internal state card using `runtime-brief.md`. Keep it mental/private. Do not print it, write it into the chat, or put it in `draft.md` unless the user explicitly asks for process notes:

- date zone and phase
- genre
- current place
- body state
- money state
- social friction
- screen/platform texture
- what is being avoided
- what hurts
- what detail is accidental and not theme-serving
- target length and rhythm constraints for the chosen validation mode
- feature budget

This state card is internal unless the user asks to see process notes. In clean-eval generation, visible process chatter contaminates the run even if the final `draft.md` is clean.

### 3. Sample Calibration

Read 3-5 short anchors, not the whole archive:

- 1-2 cards or samples from the same phase
- 1 sample from the same genre
- 1 contrasting sample to avoid monotone

Use bundled materials only during generation: `references/corpus-cards/`, `references/samples-index.md`, and `references/portable-corpus.md`. Do not require a local directory of the 38 originals for ordinary or clean-eval generation. Full originals are optional developer/controller inputs for rebuilding cards, copy-overlap checks, corpus calibration, or blind rounds when `ANLIN_CORPUS_DIR` / `--corpus-dir <corpus-dir>` is available.

For clean-eval generation, skip this step before the first draft. Draft from the minimal generation pack, run the checker, then read at most 1-2 cards only if repair is needed. The goal is to avoid long reference browsing and prompt-overfitting.

### 4. Generate Candidate Scenes

List 8-12 candidate fragments. Each fragment must come from one of:

- user-provided real material
- date/era texture
- plausible inferred daily life
- memory triggered by current sensory detail
- accidental observation

Mark each candidate by function: laugh, sting, tenderness, deflection, absurdity, analysis, body, money, social collision, memory, or noise. At least 30% should not directly serve the main theme. Keep at least two boring or accidental details available for the anti-detection pass.

Do not make background facts into candidate scenes by themselves. A valid candidate must still have an action, body effect, social change, money cost, app surface, route problem, joke, or memory trigger if named background tags are removed. Game is allowed as a candidate, but only as one of those lived functions; it is not a required Anlin marker and it is not forbidden by prompt silence.

Before drafting, run a fact gate on the selected slate: load `references/anlin-background.md` and, for machine-readable classification, `references/background-fact-classes.json` only for facts that are already present in selected scenes, then lower or remove unsupported specifics. If a new fact appears only because either background file listed it, cut it before Step 5.

### 5. Draft With Scene Modes

For clean-eval generation before the first draft, use the mode summaries already distilled in `clean-generation-brief.md`. Load `generation-modes.md` only if the scene slate is stuck, the first checker fails, or the task is ordinary non-formal drafting. Do not force every scene through one template.

Select the smallest set of scenes that can carry the piece. Standard diary usually uses several fragments; sincere and micro-hope pieces can be short. The five-step cognitive path is a strong mode for misread/self-sabotage scenes, not a global obligation.

For blind-evaluation drafts, always produce a complete article with a title. Put the generated title on the first line as plain text or `# Title`; do not bold it, label it as `标题：`, or wrap it in controller notes. Serious blind review keeps and normalizes titles for all samples, then length-matches complete articles. Title choice should follow the title model: `日寄` is a valid corpus pattern, not a universal default; `X日寄`, question, meme/platform, sentence, and literary-phrase titles are also valid when the body earns them and they do not expose the prompt. A formal standard diary should usually aim around 950-1150 body Chinese characters and preferably not by padding; below 900 is a generated-draft risk buffer, while 900-949 needs stronger scene movement and should not be treated as a standalone failure. Short sincere or micro-hope pieces require short-original matched evaluation; otherwise expand with concrete actions, dialogue, body/money consequence, and non-theme residue before blind testing.
Choosing a non-`日寄` title does not make a standard diary a short form. If the body is a standard diary, keep the same complete-article length and rhythm corridor even when the title is a side object, phrase, or sentence.
If the selected genre is sincere, do not pad it into a standard diary just to satisfy standard length. It still must be a complete article: title, concrete memory/current action, visible cost, restraint, one rough ordinary detail, and a fact-based retreat rather than a clean lesson.
Short sincere or micro-hope is not polished minimalism and not a tiny sketch. Keep a short-genre breathing corridor: when the user asks for a complete article, usually around 650-850 body Chinese characters, about 28-55 body lines, 4-7 uneven clusters, at least several longer clumsy action/speech/memory/reply lines, and one or two short fact-retreat drops earned by the scene. A 250-500-character sincere draft with no longer clumsy line is usually incomplete restraint, not a finished article; rebuild it with one present practical cluster that changes action or reply. A 520-649-character sincere draft is only a matched-short-original buffer; in clean-eval it usually needs more lived movement unless it already has real long clumsy lines and a practical turn. A 55+ line sincere draft made of mostly 4-12-character captions is a poem-shaped grid, not a stronger article; merge and rebuild before the checker. A 520+ character sincere draft can also fail if it is 8-18 smooth paragraph lines with almost every line ending in `。`; rewrite it into uneven clusters before the checker. If the user supplied several strong family-memory props, choose one main prop and at most one cropped trace; do not visibly preserve every supplied object, memory, message, and ending. For the common Mother's Day packet, do not keep both eggs and childhood rain as full scenes, do not keep both holiday feed and typed-then-deleted message as full scenes, and do not use a date/holiday title such as `五月十二日`. If the draft becomes a smooth 25-35-line prose essay with every line finishing the emotion, repair the source movement: combine adjacent memory/object facts into one rougher line, add an awkward reply or practical interruption already implied by the day, and cut the sentence that explains the feeling. Do not expand it into a standard diary just to satisfy global ratios. Do not repair short sincere profile drift by adding new food, delivery, gift, video-teaching, variety-show, game, route, or background packets; longer lines should be built from the already selected object, reply, room, body, and memory facts. For controller blind review of short sincere/micro-hope drafts, use genre-matched original/placebo rounds; do not compare a short sincere draft only against random standard diaries and call the result stable evidence.

For direct mother/holiday prompts, `cropped trace` is stricter than "write it briefly." One line with childhood rain, raincoat, broken umbrella, and school arrival is a full second memory family. One line with eggs, trip home, plastic bag, and mother's handoff is a full second object-memory family. In the common no-message + eggs + childhood-rain packet, pick one family to be visible pressure, allow at most one other to appear as a partial residue, and let the third disappear. If you keep eggs, the childhood-rain care scene should usually be absent; if you keep childhood rain, eggs should stay a present object only; if you keep no-message/holiday-screen, do not also build a feed montage and an egg/rain memory chain.

Short sincere pieces also must not become neat literary short stories. If the title object, mother/family memory, childhood rain or care scene, withheld message, and final residue all explain the same love/guilt argument, break that arc before saving: discard or bury one prompt-prop family, cut one symbolic echo, add a present practical interruption or awkward reply, and end on an unfinished action rather than letting the object prove the title.

For short sincere family or Mother's Day prompts, choose the present-action anchor before choosing the memory. The anchor is one current action that would still exist without the prompt and changes the next move: a sink or bowl problem, a door/neighbor interruption, a bad or unsent reply, a body or clothing embarrassment, a room chore, a call, or a small object failure. The first 8-12 body lines should let that current action create its own trouble before the strong prompt props arrive. If `鸡蛋`, `塑料袋`, `下雨`, `雨衣`, `朋友圈`, `母亲节`, or `没发祝福` enter in the first few lines as a neat sequence, the article is already being organized by the task prompt, even if the first sentence mentions a sink or bowl. Do not let `eggs -> childhood rain -> no blessing -> final residue` become the article's spine. The memory should leak from the action; it should not arrive first, or arrive immediately after one token action, and then recruit present objects as proof.

For short sincere titles, avoid making the prompt prop the structural key. A title like `鸡蛋`, `一袋鸡蛋`, `塑料袋`, `屏幕`, or `没发出去的消息` is high-risk when that same prop opens the body, carries the family memory, and returns at the end. Prefer a side-action or low-status handle from today's failed action. If a prompt-prop title is kept, the body must weaken it and the ending must leave through another practical action.

If a short sincere repair has already formed that memory-first spine, do not preserve the same title/object chain and merely add one current detail. Treat the old spine as spent: choose a new side-action title, start from the current practical failure, keep at most one family-memory prop as a cropped trace, and delete or bury the other proof props. This is a source reset, not a line edit.

For clean-eval standard diary generation, do not save a skeletal placeholder as `draft.md`. The first written `draft.md` must already be a complete article with title, roughly 45-70 non-empty body lines, at least six naturally longer lines, and near 950-1200 body Chinese characters when the model tends to shorten. If the candidate is visibly short, over-fragmented, or mostly 4-10 character lines, merge/expand before writing/running the checker rather than relying on a later terminal-only repair. If it is drifting beyond about 1250 characters, cut material that does not change action, social position, body consequence, or the next scene; 1350+ is an overfilled generated-draft risk, not a safer complete article. A candidate with 90+ body lines or zero long lines is not ready to write.
